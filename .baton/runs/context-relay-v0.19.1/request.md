# Request — context-relay-v0.19.1

## 사용자 의도 (명확화된 모델)

상시 양방향 대화(토큰 낭비 큼)가 **아니라**, **이벤트/요청 기반(SSE처럼)** 으로 "트리거가
와야 다음 AI가 그 작업을 한 번 진행하는" 모델. 즉:
- 역할A 완료 → 그 완료가 역할B의 **트리거** → 역할B AI가 **1회** 호출(요청→응답) → 다음 트리거.
- "통신"은 채팅이 아니라 **요청 페이로드에 실려서** 전달(이전 작업 결과를 다음 요청에 동봉).
- 진행은 이벤트 스트림으로 흐름(SSE의 그 부분 = 기존 `events.jsonl` + `baton watch`).
- AI는 트리거당 1회만 실행 → **저비용**(놀면서 토큰 태우는 상시 루프 없음).

## 현재 격차 (v0.19 대비)

v0.19 골격은 트리거(완료→다음)·스트림·1회 호출까지 이미 있음. **딱 하나 빠진 것**:
`buildRolePrompt`가 이전 역할 산출물을 **요청에 싣지 않음**(각 역할 독립 실행). 이걸 채우면
사용자가 그리던 구조가 된다.

## 이 마일스톤 범위

**컨텍스트 릴레이**: 트리거가 다음 역할을 깨울 때, 그 역할의 **상위(보고 체인) 역할** 결과를
**요약·절단**해 요청에 동봉. 단방향·이벤트 트리거 유지. **stub 위에서 headless 검증**(실제
codex/claude 디스패치는 다음, Swift 모니터는 그 다음).

토큰 효율이 1순위 제약: 전체 누적이 아니라 **보고 체인만**, 역할별 요약을 **최대 길이로 절단**,
산출물은 **경로 참조**(내용 미포함, 다운스트림이 필요 시 직접 읽음).
